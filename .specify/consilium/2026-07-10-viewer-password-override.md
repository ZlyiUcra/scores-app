# Env-override пароля сідованого viewer

## Узгоджена пропозиція

Пароль сідованого акаунта viewer перестає бути захардкодженим well-known значенням у сорсі. Оператор
задає його змінною середовища VIEWER_PASSWORD; у продакшені змінна обов'язкова (сервер відмовляється
стартувати без неї, як уже робить для JWT_SECRET і ADMIN_PASSWORD), у розробці лишається зручний
дефолт viewer123. Причина: у застосунку не існує способу змінити пароль користувача після створення,
тож сід - єдина точка, де пароль viewer взагалі можна задати; промоушен сідованого viewer до admin
назавжди лишав би повний адмін-доступ за паролем з публічного репозиторію.

## Межі та обмеження

- В скоупі: резолвер у config.ts (патерн resolveAdminPassword + мін-довжина >=8 як у registerSchema),
  заміна літерала в seedUsers, док-синхрон (комент bootstrap.ts, JSDoc config.ts, quickstart.md
  специфікації 002, README-таблиці креденшлів у 3 мовах).
- Поза скоупом: латання resolveAdminPassword (нуль вимог до довжини) - окрема пропозиція; ендпоінт
  зміни пароля; заборона промоушену сідованих акаунтів; аудит login-подій.
- Семантика сіда незмінна: лише на порожню таблицю users. VIEWER_PASSWORD - НЕ механізм ротації:
  на живій базі зміна env - no-op.

## Зауваження, що вціліли

- [підтверджено] Промоушен viewer -> admin лишає viewer123 назавжди: updateUserSchema приймає лише
  active/role, ендпоінта зміни пароля не існує ні адмінського, ні self-service (security; факт-чек
  допитувача підсилив вектор).
- [підтверджено] Навмисний fresh boot (новий сезон, порожня БД на persistent диску) - штатний
  життєвий цикл; опційний патерн тихо сідив би viewer123 у prod без жодного сигналу (security).
- [підтверджено] Throw = нова обов'язкова prod-змінна заднім числом: існуючий prod-boot після
  апгрейду впаде до виставлення env. Прийнято свідомо: чекліст уже вимагає JWT_SECRET +
  ADMIN_PASSWORD, третій рядок - той самий рух (nitpicker; ціну визнали bp і pragmatist).
- [підтверджено] Док-синхрон ширший за заявлений: bootstrap.ts:130-134 (згадує лише ADMIN_PASSWORD),
  JSDoc config.ts, specs/002-dep-upgrade/quickstart.md:11+84 (перелік обов'язкових prod-змінних і
  готова команда), README-таблиці креденшлів у 3 мовах (nitpicker, best-practices).
- [підтверджено] "First boot only, NOT a rotation mechanism" мусить стояти в доках явно - інакше
  оператор "ротує" пароль через env після витоку і вважає загрозу знятою (security).
- [підтверджено] Мін-довжина >=8 (як registerSchema): bcrypt тне на 72 байтах, словниковий перебір
  через login 10/min проходить топ-словник за години (security; користувач затвердив включення).
- [підтверджено] Env-override не лікує шерену ідентичність viewer - і не вдає, що лікує; це окрема
  властивість спільного акаунта (security, спостереження).
- [підтверджено] Перф-поверхні нема: seedUsers - boot-time once, guard на count>0; boot-порядок
  гарантовано ESM-графом (config резолвиться при import) (performance).

## Позначене допитувачем

- [слабке] Вектор вичерпаного MAX_USERS ("реєстрація відкрита" перестає знецінювати креденшл) -
  security сам понизив до другорядного: шкода там - DoS реєстрації, окрема проблема. Рішення
  користувача: прийнято як другорядне, не впливає на дизайн.
- [суперечить security] Початкові рекомендації pragmatist і best-practices (опційний патерн як
  MAX_USERS) - обидва ЗМІНИЛИ позицію під сценаріями допиту (промоушен + fresh boot): премісса
  "той самий привілей, що самореєстрація" не тримається, коли привілей за один admin-клік, а
  пароль промоушен переживає. Рішення користувача: prod-throw затверджено.

## Свідомо не робимо

- Опційний резолвер з prod-warn: fail-open з декорацією - warn ніхто не читає рівно в тих
  сценаріях, де ризик матеріалізується (формулювання best-practices після допиту).
- Видалення viewer-сіда з prod: зміна поведінки поза скоупом; дешево зробити потім (мінус один
  елемент масиву seeds), якщо буде рішення.
- Enforced-заборона промоушену сідованих акаунтів / ендпоінт зміни пароля: більший диф заради
  того самого ефекту, який дає throw у єдиній точці задання пароля.
- Латання resolveAdminPassword мін-довжиною: окрема пропозиція, потребує окремого сайн-офу
  (зачіпає існуючі деплої інакше).

## Маршрут виконання

- Рекомендація ради: direct-verified (4 з 5; performance - direct) - диф ~15 рядків не вартий
  spec-ланцюга, але сідинг не покритий тестами і торкається auth.
- Рішення користувача: підтвердив direct-verified.
- Verification-список (direct-verified):
  1. server typecheck + lint чисті.
  2. Dev-boot, temp DATA_DIR, без VIEWER_PASSWORD: viewer сідиться, логін viewer/viewer123 - 200.
  3. Dev-boot, temp DATA_DIR, VIEWER_PASSWORD задано (>=8): логін новим паролем - 200, старим - 401.
  4. Prod-boot (JWT_SECRET + ADMIN_PASSWORD задані) БЕЗ VIEWER_PASSWORD: процес відмовляється
     стартувати з чіткою помилкою.
  5. Prod-boot з усіма трьома змінними, порожня БД: сід кастомним паролем, логін - 200.
  6. Непорожня таблиця users: зміна VIEWER_PASSWORD + рестарт - no-op, старий пароль діє
     (підтверджує "not a rotation mechanism").
  7. Док-синхрон на місці: bootstrap-комент, config JSDoc, quickstart.md:11+84, README x3 мови.
