# Апгрейд залежностей клієнта і сервера

## Узгоджена пропозиція

Оновити залежності застосунку рівно настільки, щоб закрити відомі вразливості і зняти шум аудиту, не
змінюючи поведінку продукту для користувача. Обсяг "зараз" - два кроки, кожен окремим комітом. Крок 1,
серверна гігієна: підняти cookie і express-rate-limit (новий мажор лімітера робить ліміти login/register
чесними для IPv6-клієнтів - у старому їх можна було обходити ротацією адрес). Крок 2, фронтенд-тулчейн:
підняти vite до лінійки 7 - це закриває всі чотири advisory dev-сервера, включно з єдиним підтвердженим
активним вектором: витоком NTLMv2-хешу Windows через службовий ендпоінт dev-сервера, поки розробник тримає
його запущеним. Розробка ведеться саме на Windows, тому вікно реальне.

Цінність: чистий `npm audit` в обох воркспейсах, закритий активний вектор dev-машини, і зафіксований
backlog з конкретними тригерами повернення для всього, що свідомо відкладено. Решта мажорів (express 5,
zod 4, react 19 з роутером і zustand, TypeScript 7, vite 8) цінності зараз не дає і йде в backlog, а не в
роботу.

## Межі та обмеження

- В скоупі, server: `cookie` ^0.7.2, `express-rate-limit` ^8. В скоупі, client: `vite` 7.3.6,
  `@vitejs/plugin-react` ^5 (шістка вимагає vite 8 - не брати). Root: `engines.node` підняти до `>=22.12`
  (вимога vite 7; поточний `>=22.5` впускає непідтримувані версії).
- Оновлюються і діапазони в package.json, не лише lockfile.
- Два окремі коміти: (1) серверні піни, (2) vite-тулчейн. Кожен коміт підтверджується користувачем.
- Міграція vite НЕ торкається `server.fs` і `host` у vite.config.ts; якщо без цього не збирається -
  зупинитись і розбиратись, а не розширювати доступ.
- Верифікація (тестів нема, тому смоук-лист фіксований): build + typecheck обох воркспейсів; dev-режим з
  live-оновленням рахунку через socket; прод-збірка під express (NODE_ENV=production + client/dist):
  логін, deep-link на сітку, /api/health віддає JSON, неіснуючий /api/* НЕ віддає index.html, admin-імпорт
  файлу >16KB проходить, 17KB body на звичайний мутаційний роут = 413, експорт працює; 11-та спроба логіна
  за хвилину = 429 (лімітер живий після мажору); `npm ls cookie` без 0.6.0 у дереві; розмір бандла проти
  baseline 109 070 B gzip.
- Критерій відкату кроку 2 - провал будь-якого пункту смоуку; відкат = revert коміту (повернення на
  vite 5.4.21 не відкриває нових CVE понад уже наявні).
- Поза скоупом: усе з розділів backlog і "Свідомо не робимо".

## Зауваження, що вціліли

- [підтверджено] Активний вектор рівно один: NTLMv2-витік через `/__open-in-editor` (сліпий крос-сайт GET
  на UNC-шлях, middleware без Origin-перевірки, доведено з забандленого коду vite 5.4.21); закривається
  vite >=7, де launch-editor зникає з дерева залежностей (security, після допиту).
- [підтверджено] vite 7.3.6 закриває всі чотири advisory - жоден вразливий діапазон не сягає 7.x; стрибок
  на 8 (Rolldown) не потрібен для безпеки і виноситься в окремий нетерміновий крок (best-practices, з
  доказом від npm view; підтримано nitpicker і security).
- [підтверджено] cookie-пін - гігієна аудиту, не закриття активної діри: єдиний прямий виклик -
  `cookie.parse` в auth.ts:181, невразливий шлях CVE-2024-47764 (security, nitpicker).
- [підтверджено] Розчепити "cookie-пін" і "vite-тулчейн" на окремі кроки: перший - хвилини, другий -
  найбільша регресійна поверхня плану; лейбл "безпекова хвиля" на обох маскував обсяг (nitpicker).
- [підтверджено] `engines: ">=22.5"` у root package.json впускає node 22.5-22.11, які vite 7 не підтримує
  (nitpicker; перевірено проти engines vite 7.3.6).
- [підтверджено] "Ручний запуск" без фіксованого смоук-листа - ручний помах; смоук мусить включати
  прод-гілку сервера (express.static + SPA fallback живуть лише в isProd) (nitpicker, best-practices).
- [підтверджено] express-rate-limit 8 - посилення (IPv6 /56-маскування), код уже на v7-іменуванні
  (`limit`, `standardHeaders`); перевірити чистий старт з `trust proxy: 1` і реальний 429 (security,
  best-practices).
- [підтверджено] Для майбутнього кроку express 5: `app.get('*')` у index.ts:67 впаде на path-to-regexp v8,
  причому лише в прод-гілці; guard `/api` і порядок монтування зберегти; після міграції перевірити
  подвійний body-parser (незадокументований `req._body`) (best-practices, nitpicker, security - в
  backlog-картку express).
- [підтверджено] Перф-мотивації в плані немає і не повинно бути: гарячих шляхів апгрейд не чіпає, "менший
  бандл" від майбутньої react-хвилі недоведений - тому обов'язковий замір бандла до/після (performance).
- [підтверджено] Апгрейд не закриває реальні відкриті пункти проду: захардкоджений viewer/viewer123 без
  env-override (bootstrap.ts:139) і data durability (хостинг) - лишаються в backlog поруч (security,
  pragmatist).

## Позначене допитувачем

- [слабке -> посилене] "react 18 підтримуватиметься роками" (pragmatist) - на допиті визнано оцінкою без
  джерела; замінено конкретними тригерами повернення хвилі react (див. backlog). Рішення користувача:
  прийнято, хвиля в backlog з тригерами.
- [слабке -> знято автором] "RR7 дає нуль або +кілька KB" (performance) - джерела заміру нема; понижено до
  "невідомо", звідси обов'язковий baseline-замір бандла. Рішення користувача: прийнято.
- [суперечило security -> розв'язано доказами] "vite dev-CVE - не 'закрити сьогодні'" (nitpicker) проти
  "єдиний сьогодні експлуатований вектор" (security): допит зняв два з трьох векторів security (esbuild
  advisory стосується лише esbuild.serve, який vite не піднімає; крос-оріджин читання сорсів закрите
  січневими бекпортами, що вже є в 5.4.21), третій підтвердив. Рішення користувача: vite-крок робимо
  зараз, але окремим кроком і на 7.3.6, без роздутого безпекового лейбла.

## Свідомо не робимо (backlog з тригерами повернення)

- vite 7 -> 8 (Rolldown): безпекова ціль досягнута на 7.3.6; окремий крок коли-небудь, без терміновості.
- express 4 -> 5: перегляд у грудні 2026; тригер - оголошена дата EOL ближче ніж за 6 місяців АБО перший
  непофікшений у 4.x CVE рівня moderate+. Обсяг відомий: `app.get('*')` -> `'/*splat'` + @types/express 5
  + перевірка body-лімітів в обидва боки.
- zod 3 -> 4: дешево (один рядок error-API, drift-guards ловлять дрейф інференсу на typecheck), але
  цінності зараз нуль; робити разом з express 5 або за власним тригером (кінець підтримки 3.25.x);
  пам'ятати про тиху зміну текстів 400-х і негативний смоук валідації.
- react 19 + react-router 7 + zustand 5: тригери - кінець security-фіксів react 18, peer-вимога react >=19
  від потрібної залежності, або зупинка патчів react-router 6 (найімовірніший перший). Код уже готовий
  (createRoot, future-флаги RR7 увімкнені, стори на іменованому create); обов'язкова умова - замір бандла
  проти baseline.
- TypeScript 7: GA 2026-07-08, чекаємо підтвердження від typescript-eslint / tsx / @vitejs.
- bcryptjs 3 (пакування/ESM, не фікс безпеки), cookie 2.x (без CVE між 0.7.2 і 2.x), @types/node 26
  (тримати під реальну версію node на хостингу).
