# Селектор турніру в адмінці: скоуп по вкладках і вибір у URL

## Узгоджена пропозиція

Селектор турніру в шапці адмінки сьогодні показується на всіх п'яти вкладках, хоча впливає лише на дві
(Games і Squads); на решті він - мертвий контрол, що оманливо натякає на скоупінг. Прибираємо селектор
(і повідомлення про помилку фіда) з нескоуплених вкладок, а сам вибір турніру переносимо з ефемерного
стану компонента в URL: адмін може перезавантажити сторінку, поділитися лінком або відкрити закладку -
і опинитися в тому самому робочому контексті. Вибір переживає ходіння по всіх вкладках адмінки.

Заразом закривається споріднений дефект, який зміна оголила б: за порожньої бази (нуль турнірів)
уся адмінка висіла на вічному сплеші, включно з вкладкою Tournaments, де перший турнір і створюють.
Глобальні вкладки перестають залежати від існування турнірів; скоуплені за порожнього списку показують
підказку створити турнір.

## Межі та обмеження

- Вибір - query-параметр `?t=<id>` на маршрутах `/admin/*`; дерево роутів НЕ змінюється.
- URL - єдине джерело вибору: `selectedId` useState видаляється, вибір деривується з `searchParams`
  через наявний фолбек `find(t) ?? find(defaultId) ?? null`. Жодного дзеркала в стейті чи сторі
  (гібрид відхилено: popstate-дрейф і war-with-history доведені на допиті).
- Відсутній `?t` = server default; параметр не дописується, доки адмін явно не вибрав.
- Зміна селекта -> `setSearchParams({t}, {replace: true})` - вибір не засмічує back-стек;
  кліки по вкладках - звичайний push (NavLink за замовчуванням).
- Усі 5 NavLink субнавігації несуть поточний `?t` через ОДИН хелпер побудови лінка (єдиний власник
  транспорту id); косметичний `?t` на глобальних вкладках - прийнятий шум.
- Битий/зниклий `?t` -> фолбек на default + канонікалізуючий replace, що стрипає параметр: URL ніколи
  не показує id, який не відповідає екрану. Канонікалізація спрацьовує лише ПІСЛЯ завершення re-fetch
  списку, який AdminLayout вже робить на вході - інакше deep-link на щойно створений турнір мовчки
  з'їде в default.
- Сирий параметр - лише ключ exact-match `find()` по завантаженому списку; ніколи не рендериться і не
  підставляється в цілі навігації. Цілі фолбеків - константні внутрішні шляхи.
- Фід (`useTournamentFeed`) лишається резидентом AdminLayout БЕЗ гейтингу по вкладках. Інваріант:
  resolved id, що подається у фід, змінюється лише від явного вибору користувача або зникнення
  турніра зі списку - ніколи як побічний ефект навігації по вкладках. Перевірка: обхід усіх 5 вкладок
  туди-назад з не-default вибором = нуль refetch-ів і нуль реконектів сокета в Network.
- tour-bar (селектор + finished-note) і `feedError`-гілка рендеряться лише на `/admin/matches` і
  `/admin/squads`. На скоуплених write-сторінках видимий індикатор вибраного турніру обов'язковий
  (захист від deep-link плутанини скоупу).
- Deadlock порожнього списку: guard `!tournament` скоупиться разом з tour-bar; глобальні вкладки
  рендерять Outlet без AdminTournamentContext (єдиний споживач useAdminTournament монтується лише на
  скоуплених роутах); скоуплені за порожнього списку - підказка одним i18n-ключем (en/ua/pt одночасно).
- `AdminTournamentValue` конвертується з `interface` на `type` (файл і так чіпається; сусідні файли
  не чіпати).
- Жодних нових залежностей; клієнтські межі довіри не рухаються (requireAdmin на сервері - справжня
  межа, клієнтський isAdmin-гейт не змінюється).
- Верифікація: client typecheck + ручний прогін (5 вкладок туди-назад з не-default вибором + Network;
  select; reload на `/admin/matches?t=X`; deep-link; битий id -> default і стрипнутий параметр;
  порожня БД -> глобальні вкладки живі, скоуплені з підказкою).

## Зауваження, що вціліли

- [підтверджено] Deep-link плутанина скоупу: лінк з чужим `?t` приводить адміна писати не в той
  турнір; закривається видимим tour-bar на write-сторінках + канонікалізацією URL (security).
- [підтверджено] Тихий фолбек з URL = брехливий адресний рядок; канонікалізуючий replace після
  фолбеку обов'язковий (security, nitpicker).
- [підтверджено] Deadlock порожнього списку турнірів: loaded=true + tournament=null = вічний сплеш
  на всіх вкладках; фікс у скоупі цього диффа (nitpicker).
- [підтверджено] Будь-який гейтинг фіда (структурний чи через null) = teardown+rebuild (3 фетчі +
  реконект) на кожен детур між глобальними і скоупленими вкладками; не гейтити (performance,
  pragmatist, підтверджено best-practices після допиту).
- [підтверджено] Наївний гібрид стейт+URL дрейфує на popstate: екран показує X, URL каже Y, ефект
  знищує history-запис (nitpicker, pragmatist незалежно).
- [підтверджено] Інваріант I1: екран адмінки - чиста функція URL; будь-який досяжний URL відтворює
  той самий екран при reload і шерингу (best-practices).

## Позначене допитувачем

- [слабке -> переформульовано] "Усі NavLink мусять нести ?t, інакше фід смикається" (performance) -
  було умовним на URL-primary дизайн; переформульовано в дизайн-інваріант resolved id (прийнято).
- [слабке -> виправлено] "Клік по вкладці тихо скидає вибір" (nitpicker) - правдиве лише за
  pure-URL дизайну; автор сам скоригував (прийнято у виправленій формі).
- [знято автором] Гібрид стейт+URL як мінімальний зріз (pragmatist) - відкликаний після доведеного
  popstate-дрейфу; перейшов на single-source-URL.
- [знято автором] "Витік id через URL" (security) - список турнірів і так видимий кожному
  залогіненому; знання id нічого не додає.
- [суперечить performance -> розв'язано] Структурний гейтинг фіда через nested-роут (best-practices) -
  автор назвав межу власного правила, визнав її невиконаною тут і підтвердив НЕ БЛОКУЮ для
  фід-резидента без гейтингу.
- [слабке -> пом'якшено] "Path, не query" (best-practices) - пом'якшено до "path АБО чистий query
  без резолвуючого стейту з одним хелпером"; обрано друге (менший диф, роути незмінні).

## Свідомо не робимо

- Path-сегмент `/admin/t/:id/...` - перебудова дерева роутів, legacy-редіректи і async index-редірект
  заради нульової вигоди проти чистого query.
- Варіант C (пікер у кожній скоупленій сторінці) - дублювання контролу, стан однаково живе вище.
- Disabled-селектор з підказкою на глобальних вкладках - мертвий хром + три переклади за увічнення
  проблеми.
- B (групування subnav на глобальні/турнірні) - за бортом цього диффа; окрема пропозиція, якщо
  захочеться (рішення користувача 2026-07-10).
- Гейтинг фіда по вкладках - перетворює найдорожчу реакцію застосунку на побічний ефект найчастішої
  навігації; повернутися, лише якщо фід подорожчає або детури стануть гарячим циклом.
