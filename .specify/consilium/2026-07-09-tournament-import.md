# Імпорт турніру з файлу експорту

## Узгоджена пропозиція

Адміністратор може відновити або перенести турнір із раніше завантаженого файлу експорту. Завантаження файлу
в адмінці створює цілком новий турнір з усім його вмістом - групами, командами, складами гравців, розкладом
матчів з результатами і станом сітки плей-оф - точно такими, якими вони були на момент експорту. Наявні дані
при цьому ніколи не змінюються і не затираються. Невалідний, пошкоджений або незнайомий файл голосно
відхиляється, і в систему не потрапляє нічого.

Турнір прибуває з тим статусом, який мав у файлі: завершений архів лишається архівом для перегляду,
запланований - запланованим, а відновлення живого турніру після аварії одразу повертає його глядачам. Цінність:
експорт уже існує і слугує ручним бекапом; імпорт замикає петлю відновлення - файли стають придатними для
відновлення після збою, архівування і перенесення між інстансами.

## Межі та обмеження

- Скоуп v1: лише власний формат експорту (`schemaVersion === 1`), лише створення нового турніру.
- Валідація як недовірений вхід: zod-схема перевикористовує field-валідатори з `validation.ts`
  (charset-allowlist = анти-stored-XSS межа), пінована Eq-drift-guard до `TournamentExport`; `.strict()`
  наскрізь; ключі bracket проти `bracketSlotIds()`.
- Повна валідація графа ДО першого запису: внутрішньофайлова резолюція всіх посилань, унікальність id,
  доменні інваріанти (кап групи, унікальність номера гравця, `homeId !== awayId`, кап сітки 32).
- Безумовний remint усіх id одною мапою oldId->newId на всі п'ять класів посилань, включно з
  `homeOverrideId`/`awayOverrideId` у bracket. `rev` - verbatim під zod `int().min(1)`.
- Запис через новий `services/import.ts` під `withMutationLock`, лише через storage-контракти; у контракти
  додаються `createMany` для players/teams/groups як плюральні дзеркала наявного `saveMany` (одна таблиця,
  нуль крос-репо логіки). Драйверний "importGraph" заборонений.
- Route-scoped `express.json({ limit: '1mb' })` лише на роуті імпорту; глобальні 16KB не чіпати. Окремий
  rate limiter 5/min за зразком exportLimiter. Audit-запис обов'язковий.
- Політика збою посеред запису: частковий турнір лишається, помилка повертає його id, прибирає адмін руками;
  жодного авто-cleanup (конституція, принцип I).
- Статус переноситься as-is; задокументований side effect: імпорт `active`-бекапу поруч із іншим живим
  турніром перемикає default-турнір на імпортований (у сценарії відновлення це бажана поведінка).
- UI: кнопка поруч з експортом в AdminTournaments за наявним folder-split патерном; нові i18n-ключі в
  en/ua/pt одним комітом.

## Зауваження, що вціліли

- [підтверджено] Reuse charset-валідаторів validation.ts, не структурна калька shared/types (security,
  nitpicker незалежно).
- [підтверджено] Повний pre-write referential-пас; ремап включно з bracket overrides (best-practices,
  security).
- [підтверджено] Route-scoped ліміт тіла: реальний експорт ~50-200KB, глобальні 16KB відкинуть його 413-ю
  до валідації (performance - з розрахунком, security, nitpicker, pragmatist).
- [підтверджено] `createMany` замість ~230 rewrite-all персистів під синхронним локом (performance - з
  розрахунком; best-practices погодив як плоский CRUD).
- [підтверджено] Явна політика mid-import збою без мовчазного cleanup (nitpicker, best-practices, security).
- [підтверджено] Mutation lock + audit; parse+zod до входу в лок (nitpicker, security).
- [підтверджено] Import активного бекапу перемикає default - зафіксовано як задокументований ефект
  (nitpicker, best-practices).

## Позначене допитувачем

- [знято] Вимога драйверної крос-табличної транзакції (performance) - автор визнав, що атомарність не його
  парафія, а імпорт не гарячий шлях; компроміс createMany. Рішення користувача: прийнято.
- [знято] "Не транслювати повний вміст у сокети" (performance) - безпредметне: список турнірів не пушиться
  взагалі, доведено від коду.
- [суперечить security] "rev переносити verbatim - нешкідливо" (best-practices) - на допиті визнано
  помилкою: rev 0 блокує редагування матчу з адмін-UI. Розв'язано: verbatim під zod `int().min(1)`,
  закриває обидва вектори. Рішення користувача: прийнято.
- [слабке] "Другий споживач схеми - round-trip тест" (best-practices) - тестів у проекті нема; визнано
  обіцянкою і переобґрунтовано як консюмер-бік уже відвантаженого exportSchemaVersion 1. Прийнято.

## Свідомо не робимо

- Merge/overwrite наявного турніру - складно, нікому не потрібно, суперечить принципу збереження даних.
- Чужі формати і міграції версій формату - одна перевірка `schemaVersion === 1`, на незнайомій версії відмова.
- Драйверний метод importGraph / крос-таблична транзакція - справжня транзакційність приїде з Phase C
  (postgres + withTransaction), імпорт її підхопить, бо всі записи вже йдуть через контракти.
- Магія з колізіями імен (суфікси, лічильники) - адмін перейменує наявним inline-edit.
- Стрімінгові парсери, воркери, dry-run режим - обсяг файлу (десятки-сотні KB) цього не виправдовує.
